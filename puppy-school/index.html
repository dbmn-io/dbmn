Puppy School

You're about to learn Dobermann — an API client built for anyone working with APIs who believes there has to be an easier way.

There is. You've found it.

Puppy School will take you from first connection to running real batch transactions against a live API sandbox. Along the way you'll learn how Dobermann thinks, why it works the way it does, and how to get the most out of it on a real project.

By the end of this course you will know how to:

  • Connect to any REST API and retrieve data at scale
  • Load thousands of records — flat, nested, and everything in between
  • Handle errors, reprocess failed transactions, and validate data before it hits production
  • Extract and export data from any API response
  • Share your work with your team

Every lesson builds on the last. Every badge is earned.

Lesson 1 ~10 minutes

Getting Connected

Connect Dobermann to The Training Ground and run your first API requests

Step 1

Install and Sign In

Click the Dobermann icon in your VS Code activity bar.

Not installed yet? Get it from the VS Code Marketplace.

Sign in or register for a free account. Once you're in, you're ready.

Step 2

Create Your Environment

In Dobermann, navigate to Environments and create a new one with the following details:

FieldValue
NameDBMN Puppy School
Base URLhttps://api.dbmn.io/functions/v1/playground
AuthenticationDBMN

No tokens to copy. No headers to configure manually. Dobermann injects your authentication automatically at runtime — across every endpoint in this environment.

Step 3

Your First Request

Create a new endpoint. Copy the template below, then click Paste in Dobermann — it will populate everything automatically.

// Name: Get Carriers // Method: GET // Path: /reference/carriers

Hit Run. The Console opens automatically with your results.

Step 4

Reading the Console

The Console is where every response lands. It has several tabs — let's walk through the ones that matter right now.

The Completed tab shows your successful responses as a structured data table. You should see it open by default with your carrier data. But the table might look a little flat — that's because the response wraps the data in an envelope:

{ "data": [ { "scac": "BARK", "name": "BarkPost Express", "breed": "Golden Retriever", "motto": "Every package gets a tail wag" } ] }

The carrier records you want are inside that data array. To see them as proper rows and columns, click the data array in the table to expand it, then click Set as Root. The table instantly re-renders with each carrier as its own row — SCAC codes, names, breeds, mottos — all as sortable, searchable columns.

The RAW tab shows the full HTTP conversation — the exact request sent and the complete response received, with syntax-highlighted JSON. Click it now and take a look. This is where you'll go when you need to debug what actually went over the wire. You can toggle the Request and Response checkboxes to show or hide each part.

The Training Ground uses dog-themed reference data throughout. The data structures are real — GS1-aligned, EDI-compatible, foreign key validated. The carrier names are not.

Step 5

Explore the Reference Data

Create a new endpoint for each of the following. Copy each template, click Paste in Dobermann, and hit Run.

// Name: Get Locations // Method: GET // Path: /reference/locations
// Name: Get Products // Method: GET // Path: /reference/products
// Name: Get Trading Partners // Method: GET // Path: /reference/trading-partners

For each one, use what you just learned: check the Completed tab, expand the data array, and hit Set as Root to see the records as a proper table. These are your master data tables — carriers, warehouses, products, and trading partners. You will be referencing them throughout Puppy School, and they mirror exactly the kind of lookup tables you would encounter on a real implementation project.

Dobermann Philosophy

Authentication should be configured once and forgotten. In Dobermann, auth lives at the environment level — every endpoint in that environment inherits it automatically. If your session expires, Dobermann tells you before execution begins, not halfway through a large batch run.

Dig Deeper

REST APIs use standard HTTP methods to define intent — GET retrieves data without modifying it, POST creates, PUT updates, DELETE removes. The requests you just ran are GET requests — read-only calls that leave the data exactly as you found it. MDN's HTTP methods reference is worth bookmarking.

Lesson 2 ~15 minutes

Bulk Upload at Scale

Load a flat CSV file into The Training Ground at scale using Dobermann's batch engine

Step 1

Download the Inventory CSV

Download the sample inventory file: inventory-67k.csv

This file contains 67,000 inventory records — products, locations, quantities. Flat structure, one row per record. Exactly the kind of export you would get from a WMS or ERP system.

Step 2

Create the Inventory Upload Endpoint

Create a new endpoint. Copy the template below, then click Paste in Dobermann — it will populate everything automatically.

// Name: Puppy School — Bulk Inventory Upload // Method: POST // Path: /inventory // Header: Content-Type: application/json [enabled] [ { "gtin": "{{gtin}}", "sku": "{{sku}}", "description": "{{description}}", "locationGln": "{{locationGln}}", "locationName": "{{locationName}}", "quantityOnHand": "{{quantityOnHand:number}}", "uom": "{{uom}}", "status": "{{status}}" } ]
Step 3

Load Your CSV

In Dobermann, open Batch Preparation and load the inventory CSV. Map your columns to the template fields.

Step 4

Configure Threads and Run

Start with 4 threads. Hit Run and watch the batch execute.

Once complete — run it again with 16 threads. Notice the difference.

This is Dobermann running at scale. The Training Ground handles concurrent inserts — each thread processes its own batch of records independently.

Dobermann Philosophy

Dobermann was built because loading data one record at a time is not a workflow — it is a punishment. The batch engine was designed from the ground up for scale. Threads, error tracking, pause and resume — everything exists because real implementation projects demand it.

Dig Deeper

Multi-threading means sending multiple HTTP requests concurrently rather than sequentially. Each thread handles its own slice of the data independently. The speed improvement is not linear — network latency, server capacity, and connection overhead all play a role. MDN's guide to HTTP covers the fundamentals if you want to go deeper.

Lesson 2.1 ~10 minutes

Handling Errors

Deliberately trigger batch errors, identify the failures, and reprocess them

Step 1

Download the Error CSV

Download the error inventory file: inventory-errors.csv

This file contains inventory records with deliberate errors — invalid GTINs that do not exist in the reference data. When you upload this file, some records will fail FK validation.

Step 2

Run the Upload

Use your existing Puppy School inventory endpoint. Load the error CSV and run the batch.

Some records will fail. That is expected. Watch the error count in the Dobermann console.

Step 3

Inspect the Errors

Open the batch results. Review the failed records — Dobermann tells you exactly which rows failed and why.

You should see errors like:

{ "error": "Invalid reference value.", "code": "FK_VIOLATION" }
Step 4

Fix the Reference Data

Look at the errors in your batch results. You'll see two types:

  • FK_VIOLATION — a GTIN or location GLN doesn't exist in the reference tables
  • INSERT_ERROR — a value like UOM or status doesn't match the allowed values

The FK violations are fixable — you just need to add the missing reference data. The check constraint errors are bad data in the CSV itself (invalid UOM values, bad statuses, negative quantities). Those rows will fail again on reprocess, and that's expected — in real life, you'd fix the source file and re-import.

Let's fix the FK violations. First, add one product manually. Create a new endpoint using this template:

// Name: Add Missing Product // Method: POST // Path: /reference/products // Header: Content-Type: application/json [enabled] { "gtin": "99999999999999", "sku": "SKU-FIX-001", "description": "Invisible Cat Repellent", "unitPrice": 9.99 }

Run it. You've just created a product in the reference table via the API.

Step 5

Batch Load the Remaining Products

You have three more missing GTINs. Instead of creating them one at a time, let's use batch loading — the same technique you used in Lesson 2.

Convert your endpoint into a template with variables. Copy this:

// Name: Add Missing Products (Batch) // Method: POST // Path: /reference/products // Header: Content-Type: application/json [enabled] { "gtin": "{{gtin}}", "sku": "{{sku}}", "description": "{{description}}", "unitPrice": "{{unitPrice:number}}" }

Copy the CSV below, then press Ctrl+M in Dobermann to paste it directly into Batch Preparation:

gtin,sku,description,unitPrice 00000000000000,SKU-FIX-002,Quantum Fetch Ball,14.99 11111111111111,SKU-FIX-003,Self-Walking Leash,29.99 55555555555555,SKU-FIX-004,Teleportation Dog Bed,199.99

Run the batch. Three products, one execution.

Now do the same for the missing locations. Create this endpoint:

// Name: Add Missing Locations (Batch) // Method: POST // Path: /reference/locations // Header: Content-Type: application/json [enabled] { "gln": "{{gln}}", "name": "{{name}}" }

Copy this CSV and press Ctrl+M to load it:

gln,name 9999999999999,Mystery Warehouse 0000000000000,Nonexistent Depot 8888888888888,Ghost Distribution Center

Run the batch.

That's the power of template variables — any endpoint can become a batch endpoint just by replacing hardcoded values with {{variables}}.

Step 6

Reprocess the Failed Records

Go back to your failed inventory batch. Select the failed records and hit Reprocess.

The FK violation rows should now succeed — the products and locations they reference exist. Some rows with bad UOM values or invalid statuses will fail again, and that's correct — those are genuine data quality issues that need fixing at the source.

Dobermann Philosophy

Errors are not failures — they are information. Dobermann surfaces exactly what went wrong, against which record, with enough detail to fix it. The reprocess flow exists because in a real migration, fixing and rerunning individual failures is the job. Dobermann makes that precise and repeatable.

Dig Deeper

The error you saw — FK_VIOLATION — is a foreign key constraint error. It means a value in your data references a record that does not exist in a related table. Databases use foreign keys to maintain referential integrity — ensuring your data stays consistent. MDN's HTTP response codes reference covers the 400-range errors you will encounter working with REST APIs.
